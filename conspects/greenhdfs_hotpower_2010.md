## 4. Подробный конспект по разделам
### 4.1 Abstract
- Что делает этот раздел: коротко фиксирует проблему энергетической неэффективности HDFS и вводит GreenHDFS как hybrid multi-zone вариант.
- Ключевые тезисы / аргументы:
  - Классификация данных используется как главный рычаг для формирования длинных idle-периодов в Cold Zone.
  - Производительность Hot Zone предполагается сохранить за счет консолидации нагрузки на недоиспользуемых серверах.
  - Оценка ведётся по trace-driven simulation и аналитической TCO-модели.
- Важные механизмы / модель / архитектура: логическое разделение кластера на Hot/Cold зоны + переходы файлов по температуре.
- Числа, метрики, результаты:
  - 26% снижения энергозатрат за 3-месячный период симуляции.
  - Проекция: $14.6M экономии 3-year TCO и $2.4M ежегодно при масштабировании на ~38000 серверов Yahoo Hadoop.
- Что отсюда брать в диплом: формулу «temperature classification -> placement/migration policy -> measurable system effect».
- Ограничения или оговорки: результаты в основном симуляционные и модельные, не production A/B.

### 4.2 1. Introduction
- Что делает этот раздел: объясняет, почему для HDFS не работают в лоб техники scale-down, эффективные для более stateless систем.
- Ключевые тезисы / аргументы:
  - Для inactive power modes нужны длинные интервалы простоя; в HDFS они возникают плохо из-за распределения блоков/реплик.
  - Подходы с «always-on subset + secondary replicas» имеют проблемы по write-performance и усложняют DFS-логику.
  - Авторы смещают фокус с compute-placement и replica-placement на data-placement по температуре.
- Важные механизмы / модель / архитектура:
  - Ввод температуры зоны как функции требований к performance/availability и энергопотреблению.
  - Использование data locality в Hadoop: вычисления «следуют за данными» в соответствующую зону.
- Числа, метрики, результаты:
  - Масштаб Yahoo, использованный как мотивация: 170 PB и 38000 серверов.
  - Аргумент о 10-50% загрузке CPU (по внешней ссылке) как обоснование консолидации.
- Что отсюда брать в диплом: мотивационный блок и формулировку класса задач, где policy должен управлять именно размещением данных.
- Ограничения или оговорки: нет формальной оптимизационной постановки для выбора порогов.

### 4.3 2. Analysis of a Yahoo Cluster
- Что делает этот раздел: показывает на реальных HDFS-логах, что данные естественно разделяются на «быстро остывающие» и «долго спящие».
- Ключевые тезисы / аргументы:
  - Доступы к значимой доле данных имеют «news-server-like» профиль: пики сразу после создания и быстрое затухание.
  - После окончания hot-фазы многие файлы долго лежат без обращений.
- Важные механизмы / модель / архитектура:
  - Метрики жизненного цикла:
  - `FileLifeSpanCFR`: от создания до первого чтения.
  - `FileLifeSpanCLR`: от создания до последнего чтения.
  - `FileLifeSpanLRD`: от последнего чтения до удаления.
- Числа, метрики, результаты:
  - Трасса: 3 месяца, 2600 серверов, 34M файлов, 5 PB.
  - 90.26% данных имеют `CFR < 2` дней.
  - 89.61% данных имеют `CLR < 10` дней.
  - 40% данных имеют `LRD > 20` дней.
  - В steady-state около 60% данных и около 70% файлов не читались в 20-дневном окне.
- Что отсюда брать в диплом: количественное обоснование temperature-aware transitions между классами хранения.
- Ограничения или оговорки:
  - Порог 20 дней эмпирический и workload-specific.
  - Анализ выполнен на одном enterprise-кластере.
  - В тексте заявлены «four file lifespan metrics», но явно определены только `CFR`, `CLR`, `LRD`; это нужно учитывать как неполную формализацию в статье.

### 4.4 3. GREENHDFS’S HYBRID MULTI-ZONE LAYOUT (overview)
- Что делает этот раздел: задаёт системный каркас GreenHDFS и перечисляет, какие именно политики обеспечивают энергоэффект.
- Ключевые тезисы / аргументы:
  - Зона выбирается high-level placement policy на основе per-zone data-classification.
  - Файлы должны переходить между зонами адаптивно при изменении температуры.
  - Для реализации нужны изменения в HDFS control-path (в частности heartbeat handling для inactive узлов).
- Важные механизмы / модель / архитектура:
  - Две логические зоны (Hot/Cold) с разными требованиями по SLA и power behavior.
  - Прототип существует, но его реализация не раскрыта детально.
- Числа, метрики, результаты: в этом подразделе чисел нет; он задаёт дизайн и операционную логику.
- Что отсюда брать в диплом: структуру policy plane для гибридной системы хранения.
- Ограничения или оговорки: интерфейсы/тайминги контроллера и конкретные алгоритмы даны частично.

### 4.5 3.1 Hot Zone
- Что делает этот раздел: описывает слой для часто читаемых данных, где приоритет у производительности.
- Ключевые тезисы / аргументы:
  - Hot Zone обслуживает популярные данные и поддерживает строгие SLA.
  - Экономию энергии внутри Hot Zone авторы сознательно не делают приоритетом.
- Важные механизмы / модель / архитектура:
  - Chunk Server Placement policy для размещения `n` чанков по `m` серверам с целью снизить queueing delays.
  - Серверы Hot Zone предполагаются активными постоянно.
  - Базовое распределение серверов: большинство (`70%+`) в Hot Zone.
- Числа, метрики, результаты: фиксируется доля серверов Hot Zone как дизайн-параметр.
- Что отсюда брать в диплом: идею разделения policy-целей по зонам (SLA-first для hot-класса).
- Ограничения или оговорки: алгоритм chunk placement не раскрыт полностью.

### 4.6 3.2 Cold zone
- Что делает этот раздел: описывает cold-слой для редко читаемых данных, где приоритет у энергосбережения.
- Ключевые тезисы / аргументы:
  - Cold Zone допускает trade-off «ниже performance, выше energy saving».
  - Важно минимизировать wakeups и продлевать непрерывные idle-периоды.
- Важные механизмы / модель / архитектура:
  - Сервера с большим числом дисков (storage-heavy) для плотного хранения cold-данных.
  - On-demand включение серверов.
  - No-chunking: целевой доступ к данным по возможности в пределах одного сервера.
  - In-order file allocation: сначала заполняются уже включенные серверы.
  - Периодический disk scrubbing для контроля целостности.
- Числа, метрики, результаты:
  - В оценке заложен scrub: +1 день в месяц на сервер Cold Zone.
- Что отсюда брать в диплом: policy-принцип «минимизировать число активных cold-узлов и wakeups».
- Ограничения или оговорки: влияние no-chunking на latency/parallelism детально не бенчмаркуется.

### 4.7 3.3 Energy-management Policies
- Что делает этот раздел: формализует три policy-механизма, которые обеспечивают динамику системы.
- Ключевые тезисы / аргументы:
  - `File Migration Policy`: стареющие файлы переносятся из Hot в Cold.
  - `Server Power Conservation Policy`: в Cold Zone часть серверов переводится в standby/sleep.
  - `File Reversal Policy`: файл возвращается в Hot при повторном росте обращаемости.
- Важные механизмы / модель / архитектура:
  - Мера температуры файла: возраст по последнему доступу (last access age).
  - Wake-on-LAN для возврата серверов в active.
  - Масштабирование вниз сразу по CPU/DRAM/Disks, а не только по одному компоненту.
- Числа, метрики, результаты:
  - Мотивационный ориентир: idle-сервер может потреблять около 50% от peak power.
- Что отсюда брать в диплом: прямой прототип для policy-цикла в гибриде replication/EC (aging-based demotion + popularity-based promotion).
- Ограничения или оговорки: пороги миграции/возврата заданы качественно, не формально.

### 4.8 3.4 Discussion
- Что делает этот раздел: обсуждает практические последствия и граничные режимы работы.
- Ключевые тезисы / аргументы:
  - «News-server-like» профиль должен давать устойчивые периоды сна в Cold Zone.
  - Граница Hot/Cold логическая; при пиках вычислительная мощность Cold Zone может использоваться.
  - Возможен proactive wakeup через Oozie для смягчения latency пробуждения.
- Важные механизмы / модель / архитектура:
  - Амортизация wakeup latency за счёт планировщика workflow.
  - Предположение о batch-характере нагрузок снижает критичность wakeup penalty.
- Числа, метрики, результаты: количественных новых результатов нет, это design discussion.
- Что отсюда брать в диплом: практические допущения для operational политики переходов.
- Ограничения или оговорки: performance-риски описаны концептуально, не через отдельный latency benchmark.

### 4.9 4. Evaluation Results
- Что делает этот раздел: даёт количественную валидацию подхода на trace-driven симуляторе и cost-модели.
- Ключевые тезисы / аргументы:
  - Даже power management только в Cold Zone даёт значимый эффект.
  - Гибрид с storage-heavy cold-серверами улучшает TCO/емкость.
  - Частота power transitions остаётся умеренной относительно ресурса дисков.
- Важные механизмы / модель / архитектура:
  - Trace-driven simulator на 3-месячных HDFS-трассах Yahoo.
  - Power model для CPU/DRAM/HDD с режимами `active`, `idle`, `sleep`.
  - Конфигурация: 2600 серверов, 70% Hot (1820), 30% Cold (780).
- Числа, метрики, результаты:
  - Таблица мощности:
  - Hot server (2 CPU, 8 DRAM DIMM, 4 HDD): 442.7W active, 105.3W idle, 14.1W sleep.
  - Cold server (2 CPU, 8 DRAM DIMM, 12 HDD): 578.7W active, 165.3W idle, 21.3W sleep.
  - Энергия:
  - $41,607 экономии за 3 месяца для 2600 серверов.
  - Это 26% от baseline energy costs.
  - Экстраполяция: $2,432,417 ежегодно при 38000 серверах.
  - Надежность переходов питания:
  - Максимум 70 переходов питания на сервер за 3 месяца.
  - Сопоставление с ориентиром 50,000 start/stop cycles.
  - Аналитическая стоимость:
  - 3-year TCO baseline (без energy management): $15.1M для 2600 серверов.
  - В среднем активны 198 из 780 cold-серверов.
  - Оценка: ~$1M экономии 3-year TCO и +6.8 PB хранения (для 2600).
  - Масштабная проекция: $14.6M экономии 3-year TCO на всех кластерах Yahoo.
- Что отсюда брать в диплом: набор KPI для сравнения policy-стратегий (energy%, TCO, transitions, активные узлы).
- Ограничения или оговорки:
  - Это simulation + analytical model, не end-to-end production experiment.
  - Аппаратный/ценовой контекст 2010 года.

### 4.10 5. Conclusion and Future Work
- Что делает этот раздел: закрепляет итоговый вклад и открытые направления.
- Ключевые тезисы / аргументы:
  - GreenHDFS демонстрирует измеримую энергоэкономию через data-classification-driven zoning.
  - Авторы планируют усиление cold-слоя через low-cost processors, RAID alternatives и compression.
- Важные механизмы / модель / архитектура: подтверждается центральность длинных idle-интервалов как механизма экономии.

- Ограничения или оговорки: работа не реализует erasure coding внутри GreenHDFS.

## 5. Архитектура и устройство системы / метода
- Назначение системы или метода:
  - GreenHDFS строит двухзонную архитектуру поверх HDFS, где выбор зоны и переходы между зонами управляются температурой данных для снижения энергопотребления без прямого ухудшения hot-path SLA.
- Главные компоненты и их роли:
  - `High-level data placement policy`: первичный выбор зоны (`Hot` или `Cold`) для нового/переоцениваемого файла.
  - `Per-zone data-classification policies`: правила, определяющие, какие файлы считать hot/cold.
  - `Hot Zone`: производительный слой с chunk-oriented размещением и always-on серверами.
  - `Cold Zone`: энергосберегающий слой с no-chunking, in-order allocation и aggressive power policy.
  - `File Migration Policy`: переводит файл `Hot -> Cold` при росте возраста последнего доступа.
  - `File Reversal Policy`: переводит файл `Cold -> Hot` при повторной популярности.
  - `Server Power Conservation Policy`: выбирает cold-серверы для standby/sleep и управляет возвратом в active.
  - `Wake-on-LAN`: механизм активации sleeping cold-серверов.
  - `Scrubbing`: периодическая проверка блоков/подписей в Cold Zone.
- Где находятся data / parity / replicas / metadata:
  - Data размещаются в обеих зонах в зависимости от температуры.
  - В Hot Zone данные распределяются как chunked HDFS-нагрузка.
  - В Cold Zone авторы явно отключают chunking для снижения числа задействованных серверов при обращении к файлу.
  - Parity-слой (EC/RAID в эксплуатации) в данной работе отсутствует.
  - Точный replica factor по зонам и точные отличия от стандартного HDFS replication-path не специфицированы; в future work упомянуто снижение replication через RAID alternatives.
  - Явно зафиксированы только два изменения control-path: различение `inactive` и `failed` узлов в heartbeat handling и маршрутизация доступа в «правильную» зону; точная схема хранения zone-метаданных не раскрыта.
- Как проходят read / write / update / repair / migration / transition:
  - `Read`: paper требует направлять доступы в «right zone»; чтение из Cold может инициировать wake-up узла, а при превышении порога обращений применяется `File Reversal Policy` (`Cold -> Hot`).
  - `Write`: первичная зона выбирается `High-level data placement policy`; для Cold отдельно задано in-order размещение с приоритетом уже включённых узлов.
  - `Update`: отдельный протокол обновлений в paper не специфицирован; предполагается поведение в рамках стандартной HDFS-семантики.
  - `Repair`: явно описаны только heartbeat-обработка sleeping узлов и периодический disk scrubbing; детальный путь re-replication/recovery для каждого режима сна не раскрыт.
  - `Migration`: `Hot -> Cold` по aging (last access), `Cold -> Hot` по access-threshold.
  - `Transition`: cold-серверы переходят между `active` и `standby/sleep` под управлением событий доступа/миграции/scrubbing.
- Где принимаются policy-решения:
  - Выбор зоны размещения: high-level placement policy.
  - Демоция в Cold: File Migration Policy (возраст по последнему доступу).
  - Промоция в Hot: File Reversal Policy (порог обращений).
  - Переходы питания серверов: Server Power Conservation Policy.
  - Выбор целевого узла в Cold: in-order file allocation policy.
- Какие ограничения, assumptions и неясности остаются:
  - Не раскрыт полный алгоритм chunk placement в Hot Zone.
  - Не заданы строгие формулы/оптимизация для порогов migration/reversal.
  - Не дан полный write/migration controller-loop (частота переоценки, стоимость миграций, back-pressure, порядок переключения состояния данных при переносе).
  - Не описано, где и как консистентно хранится metadata о принадлежности файла к зоне в момент переходов.
  - QoS/latency-риски обсуждены, но не измерены отдельным production-бенчмарком.

## 6. Сквозные выводы по статье
- Проблема:
  - В HDFS трудно создать длинные idle-периоды на узлах, потому что данные и нагрузка распределены по всему кластеру.
- Основная идея / вклад:
  - Управлять энергией через temperature-aware размещение данных и двусторонние переходы между Hot/Cold зонами.
- Что нового относительно известных подходов:
  - Фокус не на compute consolidation сам по себе, а на data-classification-driven хранении как первичном рычаге.
  - Явная связка data lifecycle и power-state lifecycle.
- Ключевые trade-off:
  - Экономия энергии в Cold Zone достигается ценой потенциальной задержки доступа и сложности policy-контроля.
  - Производительность Hot Zone сохраняется за счёт always-on режима и консолидации.
- Главные ограничения статьи:
  - Нет реализации EC/RAID в основном дизайне.
  - Ограниченная алгоритмическая детализация ряда политик.
  - Основная валидность результатов основана на trace-driven simulation и cost projections.

## 7. Что использовать в дипломе
- Что paper даёт напрямую (можно цитировать как факт):
  - Температура данных используется для выбора зоны хранения (`Hot`/`Cold`) и для двусторонних переходов (`File Migration Policy`, `File Reversal Policy`).
  - В Cold Zone энергия экономится через no-chunking, in-order allocation и перевод части серверов в standby/sleep с Wake-on-LAN.
  - Практическая оценка показывает KPI уровня `energy saving`, `TCO`, `power transitions`, `active cold servers`.
- Что является нашей интерпретацией применимости (не прямой результат paper):
  - Замена перехода `Hot <-> Cold` на переход между схемами избыточности `replication <-> EC` в гибридной системе диплома.
  - Использование temperature-политик GreenHDFS как шаблона для orchestration re-encoding/repair в EC-контуре.
  - Перенос цели «минимизировать wakeups» на цель «минимизировать количество дорогостоящих перекодирований и миграций».
- Роль источника в дипломе:
  - `Practical system baseline` для temperature-driven orchestration и lifecycle-переходов.
  - `Related work` по policy-управлению данными и энергопотреблением в HDFS-кластере.
  - Не является теоретической основой для кодовых параметров EC (k/m, схема кодека, декодирование).
- Какие метрики / формулировки / сравнения можно опереть на этот источник:
  - Доля cold-данных в окне наблюдения.
  - Длина hot-window и cold-dormancy периодов.
  - Процент экономии энергии к baseline.
  - TCO-эффект и число активных cold-серверов.
  - Частота power transitions как reliability proxy.
- Что нельзя переносить в диплом без оговорок:
  - Абсолютные экономические оценки 2010 года.
  - Универсальность no-chunking для всех cold-нагрузок.
  - Обещания QoS без современной валидации на текущем железе и с EC-контуром.

## 9. Термины и понятия
- Data temperature: оперативная характеристика данных по частоте/свежести обращений и требованиям к SLA.
- Hot Zone: зона хранения для популярного набора данных с приоритетом производительности.
- Cold Zone: зона хранения для редко читаемых данных с приоритетом энергосбережения.
- FileLifeSpanCFR: время от создания файла до первого чтения.
- FileLifeSpanCLR: время от создания файла до последнего чтения.
- FileLifeSpanLRD: время от последнего чтения до удаления.
- File Migration Policy: правило перевода стареющих данных в Cold Zone.
- File Reversal Policy: правило возврата данных в Hot Zone при повторной популярности.
- Server Power Conservation Policy: политика выбора cold-серверов для sleep/standby и их пробуждения.
- Wake-on-LAN: механизм сетевого пробуждения узла из энергосберегающего состояния.
- Disk scrubbing: периодическая проверка блоков на диске для контроля целостности.

## 10. Итог в одном абзаце
GreenHDFS важен для диплома как практический системный baseline, где температура данных напрямую управляет размещением и энергорежимами узлов. Работа опирается на реальные HDFS-трассы Yahoo и показывает, что профиль доступа действительно допускает выделение крупного cold-класса с длительными idle-периодами. Архитектурно особенно ценен полный policy-цикл: initial placement, migration в Cold, reversal в Hot и server power-state control. Для методологии диплома полезен и набор метрик оценки: энергия, TCO, число переходов питания и число активных узлов Cold Zone. Одновременно статья имеет пределы: это в основном trace-driven simulation и аналитические проекции, а не детальный production latency study. Также в ней нет реализации erasure coding, поэтому для гибридной `replication + EC` схемы она служит архитектурной и policy-опорой, но не прямым готовым решением по кодированию.
