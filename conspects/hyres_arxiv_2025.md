# HyRES: A Hybrid Replication and Erasure Coding Approach to Data Storage

## 1. Библиографическая карточка
- ID: `hyres_arxiv_2025`
- Авторы: Daniel E. Lucani, Marcell Feher
- Год: 2025
- Тип: preprint
- Ссылка: https://arxiv.org/html/2511.00896v1

## 2. Зачем этот источник нужен для диплома
- Источник дает формальную модель гибридной избыточности (`HyRES(R,e,l,k)`) и сравнение с чистой репликацией и MDS-кодированием на единых метриках: storage cost, file loss probability, repair traffic.
- Для темы диплома он полезен как теоретическая опора по trade-off между EC и replication, особенно для обоснования, почему гибридная схема должна рассматриваться как отдельный класс, а не как частный случай одной из техник.
- Paper не предлагает temperature-aware policy и не описывает production-оркестратор переходов между режимами хранения, поэтому его роль в дипломе: `related work + theoretical foundation`, а не готовый системный дизайн.

## 4. Подробный конспект по разделам
### 4.1 Abstract
- Авторы позиционируют HyRES как гибрид replication и erasure coding, который объединяет их сильные стороны.
- Заявлены три измерения сравнения: storage cost, file loss probability, repair traffic.
- Отдельно подчеркнут учет размера сети при анализе.

### 4.2 I. Introduction
- Вводится классический компромисс: replication удобна для hot data (быстрый доступ, простой repair), но с высоким storage overhead; EC экономит storage для cold data, но усложняет repair.
- Подчеркивается, что в литературе не хватало fair-сравнения replication/EC/hybrid с учетом размера сети, числа потерь и политики размещения.
- Формулируются две цели статьи:
  - предложить семейство схем HyRES;
  - дать математический framework сравнения с replication и EC.

### 4.3 II. System Model
- Система: `N` узлов, между которыми возможна коммуникация.
- Файл: деление на `k` фрагментов; далее добавляется избыточность в зависимости от policy.
- Ограничение размещения: один узел не хранит два фрагмента одного файла.
- Модель потерь: одинаковая вероятность потери узлов при событии потерь; анализируется условная вероятность потери файла при `L` потерянных узлах.

### 4.4 III. HyRES - Hybrid Redundancy Scheme
- Даётся определение `HyRES(R,e,l,k)` и частные случаи `l=0`, `l=1`, `l=e`.
- Леммы 1 и 2 описывают worst-case число потерь фрагментов для потери файла в разных вариантах HyRES.
- Лемма 3 задает среднюю repair cost при одной потере фрагмента для ключевых вариантов.
- Теоремы 4, 5, 6 формулируют условные вероятности потери файла при `L` потерях узлов для `l=1`, `l=0`, `l=e`.
- В разделе есть remark о связи варианта HyRES с DR-MDS.

### 4.5 IV. Performance of Replication and MDS Codes
- Для replication приводятся:
  - точная комбинаторная формула (Theorem 7),
  - upper bound через вероятность необратимой потери фрагмента (Lemma 8, Theorem 9).
- Для MDS `(n,k)` приводится closed-form условной вероятности потери файла при `L` потерях узлов (Theorem 10).
- Этот раздел нужен как baseline-анализ для сопоставления с HyRES в одинаковых условиях модели.

### 4.6 V. Numerical Results
- Сравниваются HyRES `(2,2,1,10)`, replication `R=3`, Reed-Solomon `(14,10)`.
- Показаны графики для `N=30` и `N=40` и разных `L`.
- По тексту раздела HyRES показывает более низкую вероятность потери файла в рассмотренных сценариях и согласование аналитики с симуляцией.
- Обсуждается эффект размера сети на repair traffic и отмечается сценарий, где HyRES имеет меньший средний repair traffic, чем replication `R=3`, из-за меньшего числа узлов, задействованных для файла.

### 4.7 VI. Conclusions
- HyRES подается как семейство схем, обобщающее предыдущие гибридные подходы.
- Авторы акцентируют, что сравнение схем нужно делать с учетом размера сети и числа задействованных узлов.
- В future work вынесены неоднородные вероятности потерь узлов и иерархическая сеть (intra-rack/inter-rack).

## 5. Архитектура и устройство системы / метода
- Тип paper: это не полный system paper, а формализация схемы и аналитический framework.
- Компоненты метода:
  - Данные: файл делится на `k` равных фрагментов.
  - Реплики: для каждого исходного фрагмента создаются `R` копий (всего `R*k` хранимых фрагментов исходных данных).
  - Паритет: добавляются `e` MDS-coded fragments.
  - Дополнительные coded fragments: `l` линейных комбинаций над подмножествами из этих `e` coded fragments.
- Варианты схемы:
  - `l = 0`: без дополнительных комбинаций (только `R*k + e` фрагментов).
  - `l = 1`: один локально-ремонтный coded fragment.
  - `l = e`: репликация coded-фрагментов; в статье это связывается с обобщением DR-MDS.
- Размещение/узлы:
  - Сеть из `N` узлов, предполагается полносвязная коммуникация между узлами.
  - Ограничение: один узел не хранит более одного фрагмента одного файла (`N >= n` для файла).
- Потоки операций (что явно есть в paper):
  - `repair path`: при потерях HyRES приоритизирует восстановление с минимальным числом передаваемых фрагментов; при `l=1` есть локальный ремонт одного coded fragment, при множественных потерях используется декодирование.
  - `loss analysis path`: вероятности потери файла считаются условно при `L` потерянных узлах.
- Что paper явно не раскрывает как системную архитектуру:
  - Нет metadata-plane и формата/протокола метаданных.
  - Нет production-level `read/write/update` pipeline.
  - Нет механизма `migration/transition` между replica- и EC-состояниями файла.
  - Нет узла/контроллера, который runtime-классифицирует температуру данных и выбирает схему избыточности.

## 6. Сквозные выводы по статье
- Статья дает формальный язык для гибридной избыточности: `HyRES(R,e,l,k)` плюс аналитика по loss/repair.
- Главная методическая ценность: fair comparison делается не только по худшему случаю, но и с учетом сетевого контекста (`N`, число потерь `L`, размещение).
- По заявленным результатам paper, гибрид может одновременно выиграть у чистой replication по storage cost и у части альтернатив по надежности/repair в сравнимых условиях.
- Результаты чувствительны к модели: одинаковые вероятности потерь узлов и абстрактный сетевой уровень без детального production pipeline.
- Hot/cold в статье служит мотивацией; автоматического temperature-aware выбора схемы в статье нет.

## 7. Что использовать в дипломе
- Что можно брать напрямую из paper:
  - формализацию `HyRES(R,e,l,k)` и разбор случаев `l=0/1/e`;
  - метрики сравнения: storage cost, file loss probability, repair traffic;
  - идею учитывать размер сети и число потерь в fair-сравнении схем.
- Что использовать как baseline/роль источника:
  - `theoretical foundation`: для раздела с моделированием гибридной избыточности;
  - `related work`: как работа, показывающая преимущества гибридного подхода над бинарным выбором replication vs EC.
- Что является только нашей интерпретацией применимости:
  - перенос идей HyRES на temperature-aware оркестрацию жизненного цикла данных;
  - сопоставление с системными работами (`Morph`, `ER-Store`) как мост между формальной моделью и практической политикой переходов.
- Что нельзя приписывать статье:
  - готовый алгоритм классификации температуры данных;
  - production-механизм выбора/переключения redundancy scheme в runtime.

## 9. Термины и понятия
- `HyRES` - семейство гибридных схем, сочетающих replication, MDS coding и local repair.
- `MDS code` - код с максимальной дистанцией разделимости, который восстанавливает файл из минимально необходимого числа фрагментов.
- `Repair traffic` - объём данных, который нужно передать при восстановлении потерянного фрагмента или узла.
- `File loss probability` - вероятность полной потери файла при заданном числе потерь узлов.
- `Local repairability` - возможность восстановить фрагмент, не читая весь MDS-набор целиком.

## 10. Итог в одном абзаце
HyRES - это теоретический paper про гибридную избыточность, где основное содержание состоит в параметризованном семействе схем и математическом framework для их сравнения. Авторы показывают, что гибридная схема может быть лучше чистой репликации или чистого EC по storage cost, file loss probability и repair traffic. Важно, что это именно comparison paper, а не policy выбора схемы по температуре данных. Для диплома источник полезен как formal baseline и как аргумент в пользу того, что гибридную избыточность стоит моделировать явно, а не сводить к бинарному выбору replication versus EC. Если нужен temperature-aware контекст, его стоит добавлять уже как нашу интерпретацию применимости HyRES, а не как прямой вывод статьи.
