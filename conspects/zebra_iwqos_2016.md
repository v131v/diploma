# Zebra: Demand-aware Erasure Coding for Distributed Storage Systems

## 1. Библиографическая карточка
- ID: `zebra_iwqos_2016`
- Авторы: Jun Li, Baochun Li
- Год: 2016
- Тип: conference paper
- Ссылка: https://iqua.ece.toronto.edu/papers/junli-iwqos16.pdf

## 2. Зачем этот источник нужен для диплома
- Источник даёт прямой baseline по demand-aware выбору параметров erasure coding: не один статический `(k, r)`, а несколько tier-ов с разными `k` под разный спрос.
- Для темы гибридной системы replication + EC важен как обоснование самой логики «температура/спрос -> выбор избыточности», даже если сама статья остаётся внутри RS-кодирования.
- Полезен как формальная основа для policy-слоя: оптимизация рангов `k_i` при ограничении на общий storage overhead `C` и заданной отказоустойчивости `r`.
- Даёт практический материал по онлайн-адаптации (сглаживание спроса) и по стоимости переходов между конфигурациями (migration traffic), что критично для runtime-оркестрации в дипломе.

## 4. Подробный конспект по разделам
### 4.1 Abstract
- Авторы ставят задачу совместить низкий storage overhead и низкий reconstruction overhead при нестационарном и неравномерном спросе.
- Базовая идея: кодировать данные в несколько tier-ов по demand, для каждого tier выбирать параметры EC через geometric programming.
- Заявлено, что при изменении спроса Zebra перестраивается с «marginal» migration overhead.

### 4.2 I. Introduction
- Напоминается классический компромисс: replication проста и быстра в доступе, но дорога по месту; RS даёт экономию места, но реконструкция при unavailable block требует чтения `k` блоков.
- В тексте отдельно акцентированы degraded reads и сетевые/CPU издержки реконструкции, растущие с `k` (линейно по графикам Fig. 1).
- По trace из Facebook показывается сильная skewness demand: малая доля файлов горячая, остальная масса почти не читается; спрос также меняется во времени.
- Отсюда вывод раздела: «one-code-for-all» плохо подходит, нужна адаптация кодирования под demand.

### 4.3 II. Motivation and Examples
- На toy-примере сравниваются два варианта: один tier `(3,1)` для всех блоков и два tier-а, где hot-блоки идут в `(2,1)`, а cold-блоки в `(4,1)`.
- При одинаковой отказоустойчивости и сопоставимом storage overhead двухуровневый вариант снижает среднюю стоимость реконструкции; для hot-блоков в примере получается снижение overhead с 3 до 2 блоков.
- Раздел вводит ключевой тезис Zebra: число tier-ов и их параметры должны подбираться по demand, а не фиксироваться раз и навсегда.

### 4.4 III. Related Work
- Раздел противопоставляет Zebra системам, где используется один фиксированный EC-код (или статически заданные tier-ы).
- Также обсуждаются LRC/regenerating и другие repair-ориентированные коды: они улучшают ремонт, но не решают задачу demand-aware дифференциации для hot/cold.
- Вклад Zebra позиционируется как именно demand-aware assignment данных в несколько tier-ов с контролем общего storage budget.

### 4.5 IV. System Model
- Модель: `N` блоков одинакового размера, спрос каждого блока `d_i`, вектор спроса `D`, ранг/параметр каждого блока `k_i`, вектор `K`, число выдерживаемых отказов `r`.
- Предполагаются systematic RS-коды: при доступности data block чтение идёт напрямую, parity используется для реконструкции при недоступности.
- Целевая функция: минимизировать суммарный reconstruction overhead `D·K = Σ(d_i * k_i)`.
- Ограничение: общий storage overhead не больше `C` (через сумму `1/k_i` в формулировке авторов).
- Базовая постановка получается integer geometric programming; далее авторы объясняют, почему прямое решение непрактично (масштаб, integer constraint, feasibility раскладки по stripe-ам).

### 4.6 V-A. Solving Ranks Efficiently
- Сначала используется свойство монотонности оптимума: при сортировке demand более горячие данные не должны получать больший `k`, чем более холодные.
- Для снижения сложности авторы переходят от уровня блоков к уровню файлов (файл получает единый/близкий demand-профиль) и квантуют demand через параметр `t`, объединяя близкие demand в категории.
- После этого решается ослабленная (нецелочисленная) задача, затем `k_i` округляются вверх (`ceil`) для соблюдения storage constraint.
- В тексте показано, что аппроксимация близка к идеальному решению и лучше single RS в их сценариях.

### 4.7 V-B. Encoding Data in Multiple Tiers
- Tier формируется автоматически: файлы/блоки с одинаковым итоговым `k_i` попадают в один tier.
- Внутри tier данные кодируются `(k_i, r)` RS; для `k_i = 1` это эквивалентно replication с `1 + r` копиями.
- Полосы (stripes) формируются по `k_i + r` блоков, блоки stripe размещаются по разным серверам.
- Если блоков не хватает до полного stripe, временно используется `(l, r)` с `l < k_i`; авторы отмечают, что добавочный overhead от этого в больших системах мал.

### 4.8 VI-A. Deploying Zebra with Online Demand
- Чтобы не реагировать слишком агрессивно на кратковременные всплески, спрос обновляется экспоненциальным сглаживанием:
`D <- (1 - α)D + αD0`.
- `α=1` означает полную опору на последний интервал, `α→0` делает систему более инерционной.
- На hourly-обновлении в представленном эксперименте Zebra в среднем снижает reconstruction overhead на 60.8% относительно single RS.
- Авторы отдельно отмечают: лучший `α` зависит от характера workload.

### 4.9 VI-B. Data Migration with Cauchy RS
- При смене рангов возникает миграция и пересчёт parity; наивная перекодировка может быть слишком дорогой по сети.
- Zebra использует Cauchy RS и свойство подматриц Cauchy для более дешёвых переходов между `(mk, r)` и `(k, r)`.
- Вводится `kmax`: все допустимые ранги должны быть делителями `kmax`, а фактический ранг выбирается как минимальный делитель `kmax`, не меньший расчётного `k_i`.
- Такая дискретизация даёт больше «совместимых» переходов между соседними рангами и снижает migration traffic; в показанных экспериментах фиксируется trade-off между уменьшением миграции и ростом reconstruction overhead.

### 4.10 VII-A. Simulation Methodology
- Используются два Facebook trace (FB1/FB2), интервалы обновления по часу, размер блока 64 MB.
- Сравнение идёт между Zebra, single RS и схемой с двумя статическими tier-ами RS (hot/cold) как baseline.
- Метрики: average reconstruction overhead per visit, storage overhead across demand groups, migration traffic к demand traffic.

### 4.11 VII-B. Simulation Results
- Zebra систематически выигрывает у single RS по средней стоимости реконструкции, особенно на hot-данных.
- Для топ-15% наиболее востребованных файлов в тексте приведены улучшения до 53.7% и 54.9% (в двух режимах storage overhead).
- По migration traffic показано, что при использовании предложенного подхода он остаётся ограниченным долей от demand traffic (вплоть до <10% в серии экспериментов; также фигурирует уровень до 2.4% в одном из сценариев).
- Двухуровневый статический baseline уступает Zebra из-за жёсткой конфигурации tier-ов.

### 4.12 VIII. Conclusions
- Финальный вывод статьи: demand-aware многослойное кодирование позволяет одновременно улучшить реконструкцию для hot-данных и удерживать низкий storage overhead для cold-данных.
- Для динамического workload ключевым является контроль стоимости миграций; Zebra решает это через комбинацию сглаживания спроса и Cauchy-based migration.

## 5. Архитектура и устройство системы / метода
- Это не system paper с отдельными named components и data plane, а framework для выбора и перестройки EC-параметров. Поэтому архитектуру Zebra корректно описывать как control loop, а не как полноценный storage service.
- Входы метода: demand `D`, общий storage budget `C`, число tolerable failures `r`, а в online-режиме ещё коэффициент сглаживания `α` и верхняя граница ranks `kmax`.
- Центральный цикл такой: оценка demand -> решение геометрической задачи для `K` -> округление ranks -> группировка файлов с одинаковым rank в tier -> кодирование tier'ов -> при изменении demand пересчёт ranks и миграция.
- Размещение данных устроено так, что каждый tier кодируется своим `(k_i, r)` RS-code; blocks одного tier распределяются по разным servers, а `k = 1` фактически превращается в replication с `1 + r` копиями.
- Решение о выборе redundancy принимает именно optimizer Zebra, а не внешний scheduler: он минимизирует `D · K` при ограничении на storage overhead, затем переводит непрерывное решение в целочисленные ranks.
- Для online-demand Zebra хранит прошлую оценку `D` и обновляет её по формуле `(1 - α) D + α D0`, где `D0` - самый свежий demand; это уменьшает чувствительность к кратким всплескам.
- Для migration Zebra использует Cauchy RS codes как внутренний механизм более дешёвого перехода между ranks. Когда `kmax` задан, ranks ограничиваются делителями `kmax`, а новый rank выбирается как минимальный делитель, не меньший текущего `k_i`.
- Устройство migration path честно ограничено: paper описывает способ снизить network transfer при upgrade/downgrade, но не вводит отдельный metadata service, placement manager или storage controller с явными API.

## 6. Сквозные выводы по статье
- Zebra показывает, что one-code-for-all плохо работает при skewed demand: горячие данные выгоднее держать с меньшим `k`, а холодные - с меньшим storage overhead.
- Главный практический плюс метода в том, что он делает tiering не бинарным, а многоуровневым, и тем самым лучше подстраивается под реальное распределение спроса.
- По результатам статьи Zebra снижает overall reconstruction overhead, особенно на hot data, где выигрыш достигает `54.9%`; в online-режиме средняя экономия reconstruction overhead составляет `60.8%`.
- Миграция остаётся управляемой: abstract обещает, что network transfer на миграцию занимает не более `2.4%` demand traffic, а в полном sweep по workload'у авторы показывают, что он не превышает `10%` demand traffic.
- Главный trade-off метода - чем меньше `k`, тем ниже reconstruction overhead, но тем выше storage overhead; `kmax` помогает сдерживать миграцию, но делает пространство решений более жёстким.
- Ограничения источника честные: Zebra остаётся внутри RS-семейства, не строит полноценную hybrid replication + EC систему и опирается на demand как прокси температуры данных, а не на отдельную temperature model.

## 7. Что использовать в дипломе
- Использовать как baseline для тезиса, что классификация данных должна быть многоуровневой, а не только `hot/cold`.
- Использовать как формальную схему `demand -> tier -> EC parameters -> migration`, если нужно обосновать policy layer поверх storage stack.
- Использовать как источник для критерия, что стоимость перехода между EC-схемами надо учитывать отдельно от storage overhead.
- Использовать как bridge между temperature-aware storage и выбором конкретной EC-конфигурации под разные классы данных.
- Не переносить без оговорок то, чего paper не делает: здесь нет общей replication + EC архитектуры, нет универсального controller'а и нет полноценной системы метаданных.

## 9. Термины и понятия
- `Demand-aware encoding`: выбор параметров кодирования по величине спроса на данные.
- `Tier`: группа данных с похожим спросом, кодируемая одной EC-конфигурацией.
- `Rank k`: число data blocks в `(k, r)` RS-коде; чем меньше `k`, тем ниже reconstruction overhead и выше storage overhead.
- `Reconstruction overhead`: число блоков, которые нужно прочитать, чтобы восстановить недоступный блок.
- `Degraded read`: чтение данных из временно недоступного блока через reconstruction.
- `Cauchy RS code`: вариант RS-кода, который упрощает миграцию между ranks и снижает network transfer.
- `α`: коэффициент сглаживания между старым и новым demand.
- `kmax`: верхняя граница ranks, которая помогает ограничивать пространство решений и упрощать миграцию.
- `Storage overhead C`: ограничение на суммарный объём избыточности, которое задаёт допустимый budget схемы.

## 10. Итог в одном абзаце
Zebra важен для диплома как ранний и аккуратно сформулированный baseline по demand-aware tiering в распределённом storage. Работа показывает, что один фиксированный EC-код плохо подходит для данных с неравномерным спросом, и предлагает вместо этого подбирать параметры кодирования по tiers и пересчитывать их при изменении demand. Для нашей темы особенно полезны две идеи: во-первых, температура или demand должны влиять на выбор схемы хранения, а во-вторых, переходы между конфигурациями нужно считать отдельной стоимостью, а не игнорировать. Zebra хорошо обосновывает, почему hot data стоит держать с меньшим `k`, а cold data - с меньшим storage overhead, и показывает, как уменьшать migration traffic через Cauchy RS codes и smoothing demand. При этом источник остаётся внутри RS-семейства и не решает задачу гибридной replication + EC целиком, поэтому его лучше использовать как policy-baseline и как мост к более общей temperature-aware архитектуре.
